# 从 0 基础到能自己写双电机控制项目

这份文档不是给“已经会 STM32 的人”看的，而是给现在的你看的。

目标不是让你背代码，而是让你最后能做到这 3 件事：

1. 看懂当前这个项目为什么能跑起来。
2. 知道每一块代码到底在解决什么问题。
3. 下次换一个类似项目时，你能自己从头写出一个能工作的版本。

---

## 一、先别急着看代码，先理解这个项目到底在干什么

你现在这个项目，本质上是在做一个“双电机闭环控制系统”。

系统里有这些角色：

- `STM32F407VET6`：大脑，负责运算和控制。
- `AT8236`：执行层，负责把单片机给的控制信号变成电机功率输出。
- `MG540 编码器电机`：被控制对象。
- `编码器`：告诉单片机“电机现在实际转多快”。
- `VOFA`：上位机调试工具，用来看串口返回、画波形、发命令。
- `ESTOP`：急停输入，出现危险时立刻停机。

如果把整个系统说成人话，就是：

> 你在电脑上发一个命令给单片机，单片机决定该给电机多大的 PWM，电机开始转，编码器把实际速度反馈回来，单片机再继续修正输出，最后让电机尽量接近目标速度。

这就叫：

- 没有反馈时，叫“开环控制”
- 有反馈并自动修正时，叫“闭环控制”

## 二、你必须先建立的整体脑图

以后你看到任何 STM32 电机项目，都先在脑子里画这条链路：

`命令输入 -> 数据解析 -> 目标值 -> 控制算法 -> PWM输出 -> 电机运动 -> 编码器反馈 -> 再次控制`

你当前项目也完全符合这个链路。

对应到代码里，大概是：

1. 串口收到命令
2. 解析成 `cmd_left / cmd_right` 或 `target_left / target_right`
3. 定时器每 10ms 读取一次编码器
4. 如果是 PID 模式，就计算新的控制输出
5. 把输出写入 TIM1 的比较寄存器
6. 驱动板推动电机转动
7. 编码器再把速度送回来

你以后分析项目时，一定先找这 7 步，不要一上来就陷进语法细节。

## 三、C 语言部分我只讲一个：指针

你现在不需要一口气学很多 C 语言。  
如果只挑一个最影响你看懂这份工程的知识，那就是：`指针`。

你以后会发现，STM32 项目里很多“看起来难”的代码，其实难点不是算法，而是看不懂指针。

所以这一节我只做一件事：

> 把指针讲到你能在这个项目里认出来、看懂、以后自己敢写。

### 1. 先用一句人话理解指针

指针不是“神秘语法”，你先把它理解成：

> 变量所在位置的地址

你可以把内存想成一排宿舍房间：

- 变量是房间里的东西
- 地址是房间号
- 指针就是“记住房间号的纸条”

比如有一个变量：

```c
float pid_i_left = 0.0f;
```

这个变量本身存着一个值，比如 `0.0`。  
而指针存的不是这个值，而是：

> `pid_i_left` 放在内存里的位置

### 2. `*` 和 `&` 先只记这两个意思

这是最关键的两件事。

#### `&变量`

表示：

> 取这个变量的地址

例如：

```c
&pid_i_left
```

意思就是：

> 把 `pid_i_left` 这个变量的内存地址拿出来

#### `*指针`

表示：

> 通过这个地址，找到那个变量本体

例如：

```c
*i_term
```

意思就是：

> `i_term` 里面存的是一个地址，我顺着这个地址去找到真正那个 `float` 变量

你可以先死记这一句：

- `&`：取地址
- `*`：按地址找到内容

### 3. 为什么函数里要用指针

因为函数默认传参时，传进去的是“副本”，不是原变量本体。

看这个例子：

```c
void test(float x)
{
    x = 10.0f;
}
```

如果你传进去的是：

```c
float a = 1.0f;
test(a);
```

那函数里改的是 `a` 的副本，不会真的把外面的 `a` 改掉。

但如果这样写：

```c
void test(float *x)
{
    *x = 10.0f;
}
```

然后这样调用：

```c
float a = 1.0f;
test(&a);
```

这时函数拿到的是 `a` 的地址，所以 `*x = 10.0f;` 改到的就是外面的 `a` 本体。

这就是为什么嵌入式里这么常用指针：

> 因为很多函数不是只想“看数据”，而是想“修改调用者手里的变量”。

### 4. 直接用你项目里的真例子：`pid_step()`

你项目里最适合学指针的函数就是这个：

```c
static int16_t pid_step(
  int16_t target,
  int16_t measured,
  float *i_term,
  float *prev_e
)
```

这里：

- `target` 是目标值
- `measured` 是测量值
- `i_term` 是“积分项变量的地址”
- `prev_e` 是“上次误差变量的地址”

为什么后两个要用指针？

因为 PID 每算一次，都要把：

1. 当前积分值更新掉
2. 当前误差保存成“下一次的上次误差”

也就是说，这个函数不是只计算一个返回值，它还要顺便把外面的两个状态变量改掉。

### 5. 结合调用处，你就能彻底看懂

调用代码长这样：

```c
pid_out_left = pid_step(target_left, speed_left, &pid_i_left, &pid_prev_e_left);
pid_out_right = pid_step(target_right, speed_right, &pid_i_right, &pid_prev_e_right);
```

你现在要这样读：

- `target_left`：把左轮目标值传进去
- `speed_left`：把左轮实际速度传进去
- `&pid_i_left`：把“左轮积分项变量的地址”传进去
- `&pid_prev_e_left`：把“左轮上次误差变量的地址”传进去

也就是说，这一行不是单纯“算个输出”，而是：

> 让 `pid_step()` 一边算输出，一边继续维护左轮 PID 内部状态

### 6. 函数内部是怎么改外面的变量的

在 `pid_step()` 里面，你会看到这样的写法：

```c
*i_term += pid_ki * e * dt_s;
```

这句不要怕，把它翻译成人话就是：

> 顺着 `i_term` 这个地址，找到外面的积分项变量，然后在它原来的基础上继续累加

还有：

```c
d_term = pid_kd * (e - *prev_e) / dt_s;
*prev_e = e;
```

第一句是：

> 拿出上一次误差的值参与计算

第二句是：

> 把这一次误差写回去，留给下一次用

这就是指针在控制算法里的真正用途：

> 让函数能够读写“外部状态”

### 7. 再看一个你项目里的典型指针：死区补偿

这个函数也很重要：

```c
static int16_t apply_deadzone_comp(
  int16_t pwm,
  uint8_t *running_flag,
  int8_t *dir_flag,
  uint32_t *boost_until
)
```

这里传指针，是因为这个函数不只是想根据 `pwm` 算一个结果，它还要改 3 个“状态量”：

- 电机当前是不是已经在转
- 当前方向是什么
- 启动助推什么时候结束

比如：

```c
*running_flag = 1;
*dir_flag = (int8_t)sign;
*boost_until = now + PWM_START_BOOST_MS;
```

这三句的意思都是：

> 把外面的状态变量直接改掉

如果不用指针，这个函数就很难既返回补偿后的 PWM，又顺便维护这些状态。

### 8. 你项目里还有一种常见指针：把结果“写回去”

看这个函数：

```c
static int parse_two_ints(const char *s, int *out_l, int *out_r)
```

它的作用是从字符串里解析出两个整数。

这里：

- `const char *s`：输入字符串的地址
- `int *out_l`：左值输出位置
- `int *out_r`：右值输出位置

函数里面会写：

```c
*out_l = (int)l;
*out_r = (int)r;
```

意思就是：

> 把解析出来的两个结果，写回调用者给我的那两个变量里

这类写法你以后会非常常见，因为 C 语言函数只能直接返回一个值。  
如果想返回多个结果，最常见的方法就是：

> 返回值返回“是否成功”，剩下的数据通过指针写出去

### 9. 再看一个更底层的指针：读 Flash

你项目里还有这种：

```c
const uint32_t *flash = (const uint32_t *)PID_PARAM_FLASH_ADDR;
```

这里的意思是：

> 把 `0x080E0000` 这个地址，当成一个指向 `uint32_t` 的指针来用

然后就能这样读：

```c
flash[0]
flash[1]
flash[2]
```

这本质上等价于：

```c
*(flash + 0)
*(flash + 1)
*(flash + 2)
```

这里你先不用深挖指针运算，只需要知道：

> 单片机很多外设和 Flash 都是“按地址访问”的，所以指针在 STM32 里非常自然

### 10. 为什么 `const char *s` 前面有个 `const`

这个也顺手讲掉。

```c
const char *s
```

意思是：

> `s` 指向的是一段字符数据，但我不打算通过它去修改这段内容

也就是“只读输入”。

这是一种好习惯，因为它能提醒你：

- 这个参数只是拿来读
- 不应该在函数里乱改原字符串

### 11. 你现在最容易混淆的 4 种写法

#### `float x;`

普通变量，里面直接放值。

#### `float *p;`

指针变量，里面放的是某个 `float` 的地址。

#### `p = &x;`

把 `x` 的地址交给 `p`。

#### `*p = 3.0f;`

顺着 `p` 这个地址，找到 `x`，然后把 `x` 改成 `3.0f`。

这 4 句如果你能区分开，指针就已经入门了。

### 12. 以后你在 STM32 项目里看到指针，优先按这 4 类理解

#### 第 1 类：函数要修改外部变量

例如 `pid_step()`、`apply_deadzone_comp()`

#### 第 2 类：函数要返回多个结果

例如 `parse_two_ints()`

#### 第 3 类：读写某段内存、Flash、寄存器

例如 `const uint32_t *flash`

#### 第 4 类：字符串本质上也是地址

例如 `const char *line`

你只要按这 4 类去套，很多看起来难的代码会立刻简单很多。

### 13. 你现在先不用学的指针内容

为了不把你绕晕，下面这些先别急着深挖：

- 多级指针
- 函数指针
- 复杂数组指针
- 野指针和动态内存分配细节

你当前项目最需要的，只有：

1. 会看 `&变量`
2. 会看 `*指针`
3. 知道为什么函数参数里要传地址
4. 能看懂函数里如何通过地址改外面的变量

先把这 4 个练熟就够了。

### 14. 给你的最小练习

你现在可以自己做 3 个小练习。

#### 练习 1

自己解释这句：


要求你能用人话说出每个参数在干什么。

#### 练习 2

要求你能说清：

### 15. 你什么时候算真的会了指针

当你能做到下面这几件事，就说明你已经过了“最难的心理关”：

## 四、STM32 项目最重要的思想：不是“写代码”，而是“调外设”

你以后要彻底改掉一个误区：

> STM32 项目不是像普通电脑程序那样只写算法，而是“配置外设 + 写控制逻辑”。

你这个项目主要用了 5 类外设：

- `GPIO`
- `TIM1`
- `TIM2`
- `TIM4`
- `TIM3`
- `USART3`

每一个外设都有明确分工。

### 1. GPIO：通用输入输出

这里用到了一个重要 GPIO：`ESTOP`

它被配置成外部中断输入。

意思是：

> 一旦急停信号触发，不需要主循环慢慢轮询，单片机会立刻进入中断处理。

这就是急停必须快响应的原因。

### 2. TIM1：负责 PWM 输出

TIM1 在这里不是拿来计时，而是拿来输出 4 路 PWM。

四路 PWM 的分工是：

- 左电机正转
- 左电机反转
- 右电机正转
- 右电机反转

所以你会看到代码里定义了：

- `LEFT_FORWARD_CH`
- `LEFT_BACKWARD_CH`
- `RIGHT_FORWARD_CH`
- `RIGHT_BACKWARD_CH`

### 3. TIM2、TIM4：负责编码器模式

它们的作用不是输出，而是“数脉冲”。

编码器转动时会输出 AB 两相信号，定时器在编码器模式下会自动帮你计数。

你不需要手动一个一个脉冲去数，这就是硬件外设的价值。

### 4. TIM3：负责控制节拍

TIM3 每 10ms 进一次中断。

这 10ms 对整个闭环系统非常重要，因为它定义了：

- 多久测速一次
- 多久跑一次 PID
- 多久更新一次输出

你可以把它理解成控制系统的“心跳”。

### 5. USART3：负责串口通信

它负责两件事：

1. 接收你从 VOFA 发来的命令
2. 把状态、波形数据发回给电脑

## 五、当前工程是怎样启动起来的

在 `main()` 里，系统是这样启动的：

1. `HAL_Init()`：初始化 HAL 库和基础环境
2. `SystemClock_Config()`：配置时钟到 168MHz
3. `MX_GPIO_Init()`：初始化 GPIO
4. `MX_TIM1_Init()`：初始化 PWM 定时器
5. `MX_TIM2_Init()`、`MX_TIM4_Init()`：初始化编码器定时器
6. `MX_USART3_UART_Init()`：初始化串口
7. `MX_TIM3_Init()`：初始化 10ms 控制定时器
8. `HAL_TIM_PWM_Start()`：真正开始输出 PWM
9. `HAL_TIM_Encoder_Start()`：真正开始读取编码器
10. `HAL_TIM_Base_Start_IT()`：真正打开 TIM3 中断

这里你一定要知道一个概念：

> `MX_TIM1_Init()` 不等于 PWM 已经开始输出。  
> `Init` 是配置，`Start` 才是运行。

这在 STM32 里非常常见。

## 六、PWM 是什么，你必须彻底懂

PWM 是“脉冲宽度调制”。

你现在先不要想太复杂，就记一句：

> PWM 通过“高电平持续时间占一个周期的比例”来等效改变输出能量。

例如：

- 10% 占空比：能量小
- 50% 占空比：中等
- 90% 占空比：能量大

### 1. ARR 和 CCR

在 STM32 定时器输出 PWM 时，你最常关心两个值：

- `ARR`：周期上限
- `CCR`：比较值

如果：

- `ARR = 4199`
- `CCR = 2100`

那占空比大约就是：

`2100 / 4199 ≈ 50%`

你的代码里 `PWM_MAX = 4199`，本质上就是拿 ARR 的范围做统一上限。

### 2. 为什么有正负 PWM

电机不是只能“快慢”，还要“方向”。

所以你项目里的设计不是一个通道直接输出正负，而是：

- `pwm > 0`：正转通道输出，反转通道清零
- `pwm < 0`：反转通道输出，正转通道清零
- `pwm = 0`：两个通道都为 0

这是 H 桥驱动非常常见的控制方法。

### 3. 为什么要做死区补偿

你调试时已经发现：

- 小 PWM 不转
- 起步需要更大 PWM
- 转起来之后保持运动所需 PWM 会更小

这不是代码错误，而是电机真实物理特性。

原因包括：

- 静摩擦
- 驱动压降
- 负载阻力
- 电机启动转矩需求更高

所以代码里做了：

- 最小启动值
- 最小运行值
- 起步 boost 持续一小段时间

这说明你已经开始从“写程序”进入“做控制系统”了。

## 七、编码器测速到底是怎么回事

很多初学者最容易混淆：

> 为什么 `SPD=23` 不是 23rpm？

因为代码里测到的不是 rpm，而是：

> 每 10ms 内，编码器计数增加了多少

也就是：

`counts / 10ms`

当前做法是：

1. 读取当前编码器计数器值
2. 减去上一次保存的值
3. 得到这 10ms 内的增量

增量越大，说明转得越快。

### 为什么不直接算 rpm

因为控制周期就是 10ms。

系统每 10ms 需要立刻知道速度，用这个增量最直接，计算也最简单。

如果你要显示 rpm，可以额外再换算，但闭环控制本身并不一定需要用 rpm 作为内部单位。

### 以后你要养成的习惯

每次看到速度变量，都先问自己：

1. 这个速度的单位是什么？
2. 是角速度、rpm、线速度，还是计数增量？
3. 是“瞬时值”还是“一个周期内的差值”？

不先搞清单位，控制系统很容易越调越乱。

## 八、PID 是什么，为什么这个项目需要它

你可以先把 PID 理解成：

> 一个自动修正器

如果实际速度没有达到目标速度，它就会自动增加输出；如果超了，它就会减小输出。

### 1. P：比例

误差大，就多补；误差小，就少补。

这是最直观的一项。

### 2. I：积分

如果长期都差一点点，P 可能补不够，I 会把这些“长期积累的误差”慢慢加起来，最后把系统推到目标附近。

所以 I 常用来消除稳态误差。

### 3. D：微分

它看的是“误差变化有多快”。

如果系统变化太猛，D 会起到提前刹车的作用。

但电机测速通常会有噪声，D 很容易把噪声放大，所以很多时候一开始用 0 是正常的。

### 4. FF：前馈

这是你项目里非常有价值的一项。

它不是看误差，而是：

> 目标一来，先按经验给一个大致合理的基础输出

例如目标速度越大，先给的基础 PWM 也越大。

这样 PID 后面的修正负担会更小。

### 5. PID 的本质不是公式，而是这个思维

每个控制周期做一件事：

1. 看目标是多少
2. 看实际是多少
3. 算误差
4. 根据误差决定下一拍该给多大输出

以后无论你做速度环、位置环、温度控制、舵机控制，本质上都是这个思路。

## 九、为什么中断不能乱写

这是你以后写 STM32 项目时必须建立的工程习惯。

### 1. 什么是中断

中断可以理解为：

> 程序正在干别的事情时，硬件突然说“你先来处理我”

比如：

- TIM3 到时间了
- 串口收到数据了
- 急停信号触发了

这时程序就会先去执行对应的回调函数。

### 2. 为什么中断里不要做太多事

因为中断优先级高，执行期间会影响系统实时性。

如果你在中断里：

- 长时间发串口
- 做复杂字符串解析
- 做很多浮点运算

就容易导致：

- 丢中断
- 串口异常
- 系统卡顿
- 时序变乱

### 3. 当前项目的正确做法

你这份代码用了一个很好的设计思想：

> 中断里只做“通知”，主循环里做“真正处理”

例如：

- TIM3 中断里只把 `control_tick = 1`
- 串口回调里只把完整命令缓存起来，再把 `uart3_cmd_ready = 1`
- 主循环看到标志位后，再去做 PID、字符串解析、串口回复

这是你以后写所有嵌入式项目都应该优先采用的结构。

## 十、串口协议为什么要自己设计

很多初学者觉得串口就是“随便发个数字”。

但真正工程里，串口是“协议”。

你当前项目已经有一个小协议：

- `1`：测试通信
- `PID=1`：打开闭环
- `PID=0`：关闭闭环
- `PLOT=1`：开启画图模式
- `PLOT=0`：关闭画图模式
- `STATUS`：查询状态
- `KP=400`：修改参数
- `30,0`：发送左/右目标

你要理解：

> 协议不是为了“显得专业”，而是为了让设备可控、可调、可测试。

以后你写项目时，哪怕只有自己用，也建议至少设计出：

1. 查询状态命令
2. 设置参数命令
3. 控制命令
4. 保存参数命令
5. 调试输出命令

这样调试效率会高很多。

## 十一、为什么要做串口缓冲和容错

串口不是你想象中的“每次都刚好收到一整句”。

真实情况可能是：

- 一次收到半句
- 一次收到两句
- 带回车换行
- 带空格
- 带杂讯

所以代码里要做：

1. 接收缓冲
2. 按换行组包
3. 长度保护
4. 命令裁剪
5. 数字解析容错
6. 异常重启接收

这也是为什么 `parse_two_ints()` 不是最简单的写法。

你以后写串口程序，千万不要天真地假设：

> 我每次都能一次收到完整、干净、标准的字符串

真实项目里，这个假设往往是错的。

## 十二、为什么参数要写入 Flash

如果 PID 参数只存在 RAM 里，那么：

- 一掉电就丢
- 一复位就丢

这显然不适合调参数。

所以项目里做了参数持久化：

1. 上电时从 Flash 读参数
2. 调好参数后执行 `SAVE`
3. 把参数写入指定 Flash 扇区
4. 下次上电重新加载

你要记住这个工程思想：

> 会变化、需要保存的“配置参数”要和“运行时变量”分开。

以后你做别的项目，比如：

- 舵机零点
- 传感器校准值
- PID 参数
- 通信地址
- 限位参数

这些都很适合放入 Flash。

## 十四、以后你要怎样自己从 0 写一个类似项目

这一节最重要。

你以后的目标不是只会改这一个项目，而是会“搭框架”。

下面是非常实用的开发顺序。

### 第 1 步：先写最小可运行版本

最小版本不要一开始就上 PID。

第一版只做：

1. PWM 能输出
2. 电机能正转反转
3. 串口能收 `left,right`
4. 能用 `0,0` 停车

如果第一步都没通，就不要急着上编码器和 PID。

### 第 2 步：再加编码器测速

第二版只做：

1. 编码器计数能变化
2. 能周期性算出速度增量
3. 串口能打印 `speed_left/speed_right`

注意：先验证“测得出来”，再考虑“测得准不准”。

### 第 3 步：再加定时控制节拍

第三版做：

1. 定时器固定 10ms 进一次中断
2. 在主循环按节拍测速和更新输出

这样系统开始有稳定控制节拍。

### 第 4 步：最后再加 PID

第四版再做：

1. 定义 `target`
2. 写 `pid_step()`
3. 用 PID 输出替代原来的直接 PWM 命令

这是正确顺序。

很多初学者失败，就是因为一开始就把所有功能一起上，出问题后根本不知道是哪一层坏了。

### 第 5 步：加保护和调试接口

等系统能跑了，再逐渐补：

- 急停
- 参数保存
- `STATUS`
- `PLOT`
- 串口异常恢复

这时你就进入“工程化”阶段了。

## 十五、如果你的目标是快速上手开发，就按这套方法做

这一节不再偏“解释概念”，而是偏“你拿到项目后到底该怎么开工”。

你以后开发速度能不能快起来，核心不在于你会不会很多语法，而在于你有没有一套固定开发动作。

### 1. 第一天不要碰 PID，先把基础链路打通

你第一天只做这 4 件事：

1. 工程能生成、能编译、能下载
2. 串口能收发
3. PWM 能输出
4. 电机能开环正转、反转、停止

这一步的目标不是“做完整”，而是先确认：

> 板子、驱动、外设、代码主链路都活着

### 2. 永远按这个开发顺序来

以后做类似项目，固定按这个顺序：

1. 先确认硬件接线
2. 再确认外设配置
3. 再确认最小输出
4. 再确认最小反馈
5. 最后再上控制算法

你可以把它记成一句话：

> 先让系统有反应，再让系统变聪明

### 3. 每加一层功能，马上做最小验证

这是最快上手的关键。

比如：

- 加了 PWM，就先测引脚波形
- 加了串口，就先测 `1 -> 2`
- 加了编码器，就先看计数会不会变
- 加了 PID，就先看目标变大时输出会不会跟着变大

你以后不要再“一口气写很多，再一起测”，那样最慢。

正确方式是：

> 写一点，测一点；测通了，再往下走

### 4. 每个模块都先问自己 3 句话

这是你以后最快的开发习惯。

每写一个模块，先写下：

1. 它的输入是什么？
2. 它的输出是什么？
3. 我怎么证明它现在是好的？

举例：

- `Motor_SetLeft()`  
  输入：`pwm`  
  输出：TIM1 两个通道的 CCR  
  验证：示波器或万用表能看到一个通道有 PWM，另一个为 0

- 编码器测速  
  输入：编码器 AB 相脉冲  
  输出：`speed_left/speed_right`  
  验证：手转电机时速度值变化，正反方向符号正确

### 5. 你以后开发时，优先保留这几个“调试抓手”

很多人写程序写到后面越来越难改，是因为没有留调试接口。

你以后做项目，尽量一开始就保留这些东西：

- `STATUS`
- 停机命令
- 手动 PWM 命令
- 参数查询命令
- 简单的打印或波形输出

这些东西会让你开发速度直接快一倍，因为你不会一直“靠猜”。

### 6. 先做最小系统，不要一开始就追求漂亮架构

你现在最重要的是能独立做出来，不是先把代码写得多优雅。

所以你以后开发时，先接受这件事：

> 第一版代码只要结构清楚、能测、能改、能查，就是好代码

等它稳定跑起来以后，再慢慢优化结构。

### 7. 我建议你以后固定照这个节奏做

#### 第一步：先写流程图

哪怕只写 5 行：

1. 串口收命令
2. 解析目标值
3. 定时测速
4. 算控制输出
5. 输出 PWM

先把这 5 步写出来，代码就不容易乱。

#### 第二步：CubeMX 只开当前真需要的外设

比如当前双电机项目，最小需要的就是：

- PWM
- 编码器
- 串口
- 一个基础定时器

别一上来把一堆外设都开了。

#### 第三步：先打通开环

这是最快见成果的一步。

只要开环通了，你就已经证明了：

- PWM 有输出
- 驱动能响应
- 电机能转
- 串口命令能改输出

#### 第四步：再加反馈

确认编码器计数、速度计算都正常。

#### 第五步：最后上闭环

等开环和反馈都没问题了，再上 PID。

这样做虽然看起来朴素，但实际上最快。

### 8. 给你一套最实用的排错顺序

#### 电机不转时

先查：

1. PWM 引脚有没有波形
2. 驱动输入端有没有波形
3. 电机供电是否正常
4. 共地是否接好
5. PWM 是否低于起步阈值
6. 是否被急停或停机逻辑拦住

#### 串口没反应时

先查：

1. 波特率
2. TX/RX 是否接反
3. USART 中断是否开启
4. 回调是否真的进入
5. 接收是否重新挂上

#### PID 没效果时

先查：

1. `PID=1` 是否真的开启
2. `target` 是否真的写进去了
3. `speed` 是否真的在变化
4. `pid_step()` 是否真的被调用
5. 输出是否被限幅或覆盖

你以后一旦学会“按顺序排错”，开发速度会快非常多。

### 9. 以后每个新项目，我建议你都先做一个最小版本

比如你以后再做一个控制项目，不管是什么对象，先只做：

1. 能输入命令
2. 能输出控制量
3. 能读回反馈
4. 能打印状态

只要这个最小版本通了，后面再加 PID、滤波、保护、参数保存都会容易很多。

### 10. 你现在最该练的，不是“记代码”，而是“搭步骤”

真正能快速上手的人，脑子里都有固定节奏：

- 先做哪一步
- 每一步怎么测
- 出问题先查哪一层

所以你以后别再问自己“这段代码背下来没有”，而要多问：

> 如果让我重新做一遍，我知道第一步先做什么吗？

如果你能回答这个问题，你就开始具备独立开发能力了。

## 十七、你现在最容易踩的坑

### 1. 不搞清单位

比如把：

- PWM
- 编码器增量
- rpm
- 目标速度

混在一起，这是调试时最危险的坑。

### 2. 一上来就怀疑代码，不先分层排查

正确顺序应该是：

1. 硬件接线是否正确
2. PWM 是否真的输出
3. 驱动是否真的响应
4. 编码器是否真的有反馈
5. 串口是否真的稳定
6. 最后再看 PID

### 3. 中断里做太多事

这会导致系统不稳定。

### 4. 不加限幅

没有限幅的控制输出很容易出问题。

### 5. 不做状态查询

如果没有 `STATUS` 这种命令，调试会非常痛苦。

## 二十三、给你的最终目标

你现在的目标不是成为“会改别人代码的人”，而是逐步成为：

> 能自己搭框架、自己查问题、自己写控制程序的人

当你真正做到下面这些事时，就说明你已经跨过去了：

1. 能自己画出项目结构图
2. 能自己说清每个变量的含义
3. 能自己写最小 PWM 驱动
4. 能自己写编码器测速
5. 能自己写最简单的 PID
6. 能自己设计串口调试协议

做到这里，你就已经不是 0 基础了。

---

## 二十四、我建议你接下来继续这样学

你可以把下一轮学习分成 3 个主题：

1. 只学 `main.c` 的执行流程
2. 只学 PWM 和编码器
3. 只学 PID 和调参

不要一次吃太多。

如果后面你愿意，我可以继续陪你做下面这三件事中的任意一个：

1. 带你逐段精讲 `main.c`，一行一行拆开
2. 带你手写一个“最小开环双电机程序”
3. 带你从 0 学会 PID，并且教你怎么调参数
